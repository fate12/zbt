# 主播通本地运维技能（给非开发人员用）

这个技能让不会写代码的人，也能在本地把「主播通」跑起来、改页面文字、然后把改动保存上传。

> 本文件写给两个读者：**你（使用者）** 看得懂每一步在干什么；**AI 助手** 严格照着执行。
> 原则：拿不准就先问你，绝不自己乱来、不做有风险的动作。

---

## 先懂几个词（大白话术语表）

| 词 | 大白话 |
|---|---|
| 项目根目录 | 你下载/克隆下来的「主播通」文件夹的最外层那一层（里面能看到 `apps`、`services`、`package.json`） |
| 依赖 | 项目要用的第三方软件包，第一次需要「安装」一下才能跑 |
| 启动服务 | 把网站和接口在本地运行起来，你能在浏览器里打开看效果 |
| 端口（如 5174） | 一扇门牌号，浏览器靠它找到对应的服务 |
| 改动 | 你（或助手）改过的、还没保存上传的文件内容 |
| 提交（commit） | 给改动写一句话说明，正式「存档」 |
| 推送（push） | 把存档上传到远程仓库（团队的代码库），别人就能看到 |
| 发版 | 本技能里指「提交 + 推送」一气呵成 |
| 撤销 | 把**还没提交**的改动扔掉，恢复成上次的样子 |

## 操作一：初始化环境 ｜ 关键字 `初始化`

**什么时候用**：第一次用、换了电脑、或提示「依赖未安装」时。
**你要提供什么**：项目文件夹的位置（第一次会被问到一次，之后会记住，不再问）。

### 第 0 步（按需）：克隆项目 ｜ 关键字 `克隆项目`

> 只有当**电脑上还没有这个项目**时才需要。如果已经有了，直接跳到「第 1 步」。
> 也可以单独输入「**克隆项目**」（或「下载项目 / 下载代码」）主动触发。

项目代码仓库地址：`https://github.com/fate12/zbt.git`

流程：
1. 先确认电脑上是否已经有了（看第 1 步的定位逻辑、或直接问用户）。**已经有了就不要重复克隆**，避免覆盖。
2. 问用户放在哪（给个建议默认值，让非开发人员好选）：
   > 「要把项目下载到哪个文件夹？没特殊要求的话，建议放在桌面：`~/Desktop/zbt`。」
3. 目标路径如果**已存在且非空**，停下来问用户：「这个文件夹已经有东西了，要不要换个位置？或先清空？」**不要**直接覆盖。
4. 执行克隆：
   ```bash
   # 把项目从 GitHub 下载到用户指定的文件夹
   git clone https://github.com/fate12/zbt.git "<目标路径>"
   ```
5. 克隆完成后，**把路径记下来**（写入 `~/.zhibotong-devops.root`），这样后续初始化/启动服务都能自动找到，不用再问：
   ```bash
   echo "<目标路径>" > ~/.zhibotong-devops.root
   ```

**异常处理**：
- 提示找不到 `git` → 告诉用户先安装 Git（macOS：装 Xcode Command Line Tools `xcode-select --install`）。
- 克隆失败、报权限/认证错误（仓库可能非公开）→ 把报错原样告诉用户，提示需要 GitHub 账号授权或联系开发。
- 网络超时 → 提示检查网络后重试。

克隆成功后，进入第 1 步（此时项目文件夹已确定，直接用刚克隆的路径）。

### 第 1 步：找到项目文件夹

策略是 **「问一次 → 记住 → 以后自动用」**，不会每次都问你，也不会满硬盘乱搜（慢、容易找错）。

1. 先看有没有存过路径（记录文件：`~/.zhibotong-devops.root`）：
   ```bash
   # 读取之前记下的项目路径
   [ -f ~/.zhibotong-devops.root ] && cat ~/.zhibotong-devops.root
   ```
   如果有，再确认那个文件夹还在、且是主播通项目（里面有 `pnpm-workspace.yaml`）：
   ```bash
   ROOT=$(cat ~/.zhibotong-devops.root)
   [ -f "$ROOT/pnpm-workspace.yaml" ] && echo "OK $ROOT" || echo "INVALID"
   ```
   还在 → 直接用，进入第 2 步。

2. 没存过 / 失效了 → **问你**：
   > 「请把主播通项目文件夹的路径发给我；或者直接把那个文件夹**拖进对话框**也行。」
   - 你给的路径要是最外层文件夹（里面有 `apps`、`services`、`package.json`）。
   - 如果助手说「这看起来不是项目根目录（缺少 pnpm-workspace.yaml）」，说明找错层了，重新发最外层那个。

3. 确认对了之后，**记住路径**，下次免问：
   ```bash
   echo "<项目路径>" > ~/.zhibotong-devops.root
   ```

### 第 2 步：切换到 Node 20（项目要求 20.x）

> ⚠️ 重要：悟空启动的 shell 可能和你终端不是同一个 Node（常见情况：你终端是 nvm 装的 v24，悟空是非交互 shell 用的系统 v22）。所以**不能只 `node -v` 看一眼**，必须**主动加载 nvm 并切到 20**，否则后续装依赖/跑服务会因版本不符报错。

每条命令开头都要先加载 nvm（非交互 shell 默认不会自动加载），再切版本：

```bash
# 1) 加载 nvm（如果装了的话）
export NVM_DIR="$HOME/.nvm"
[ -s "$NVM_DIR/nvm.sh" ] && \. "$NVM_DIR/nvm.sh"

# 2) 没装过 20 就装，已装会自动跳过
nvm install 20.19.0 --latest-npm

# 3) 切到 20.19.0
nvm use 20.19.0

# 4) 确认
node -v        # 必须显示 v20.19.0
```

判断与处理：
- `node -v` 显示 `v20.19.0` → 通过，进入第 3 步。
- 若机器**没装 nvm**（`$NVM_DIR` 不存在、`nvm` 命令找不到）→ 停下来告诉用户：「这台电脑没装 nvm，无法自动切换 Node 版本。请先安装 nvm 并 `nvm install 20.19.0`，或手动把 Node 切到 20.x 后再试。」**不要**用 v22/v24 凑合继续。
- 切换失败（网络等原因装不上 20）→ 把报错原样告诉用户，等处理。

> 小提示：后续每个新开的 shell 里执行 pnpm 命令前，都要先跑上面的「加载 nvm + `nvm use 20.19.0`」，否则会回落到系统默认的高版本 Node。

### 第 3 步：安装依赖（按需，不是每次都装）

```bash
corepack enable && corepack prepare pnpm@9.4.0 --activate   # 确保用对版本的工具
pnpm -v                                                     # 应显示 9.4.0
```
- 第一次（项目里没有 `node_modules` 文件夹）→ 执行 `pnpm install`，会告诉你「首次安装，可能要几分钟」。
- 之前装过、且没人动过依赖清单 → 跳过，省时间。

**完成后你会看到**：「项目目录：XXX；Node 版本：XXX；依赖就绪 ✅」。

## 操作二：启动服务 ｜ 关键字 `启动服务`

**用途**：启动本地服务用于调试修改——在浏览器里看效果、试改动。
**你要提供什么**：一般不用提供什么，直接说「启动服务」即可。
**前置**：需要先做过「初始化」（项目目录已找到、依赖已装）。没做过会先让你初始化，或先快速定位文件夹。

助手会在**后台**同时跑起 3 个服务（互不影响、保持运行），并给你可以点的网址：

| 你常说的 | 服务名 | 启动命令 | 打开网址 |
|---|---|---|---|
| AI助手 / 用户端 / 主播通前端 | user-web | `pnpm dev:user` | http://localhost:5174 |
| 后台管理 / 管理后台 | admin-web | `pnpm dev:admin` | http://localhost:5173 |
| 接口 / 后端 / API | api | `pnpm dev:api` | http://localhost:9000 |

**启动顺序**：先 API，再两个前端。因为前端要靠 API 才能正常显示数据。

**怎么算成功**：
- 接口：打开 http://localhost:9000/api/health ，看到 `{"status":"ok",...}`。
- 前端：打开对应网址，能看到登录页就行。

**端口被占用**（提示「port already in use」之类）：先确认是不是已经跑着了，避免重复启动；要停掉旧的再起，会先告诉你。

**想停掉**：说「停掉服务 / 停服」，助手会先问你是「全停」还是「停某一个」，再关对应的后台进程。

## 操作三：修改页面文案 ｜ 关键字 `改文案`

**什么是「文案」**：页面上给人看的**中文文字**——标题、按钮、提示语、说明、占位符。
**什么不算文案**（不要碰）：代码逻辑、接口地址、样式、变量名。

**你要提供什么**：哪个页面、哪句话、改成什么。
- 说得越具体越好，比如「把 AI 助手登录页的标题『欢迎回来』改成『欢迎登录主播通』」。
- 说得模糊（如「把那个按钮改一下」但没指哪个），助手会**先追问清楚**，不会瞎猜。

**助手怎么做**：
1. 搜到那句话的位置，把「当前文案」给你看，跟你确认是这一处。
2. 只改这一个字符串，不动周围任何代码。
3. 改完列清单：「改了哪几个文件、原文→新文」，交给你决定要不要「发版」。

**规矩**：一次只改你指定的，不顺手"优化"别的文字；全程中文、跟现有风格一致。

## 操作四：提交代码 或 撤销修改 ｜ 关键字 `发版` / `撤销`

改完文案后，二选一：

- 想**保存上传** → 说 **`发版`**（提交 + 推送）。
- 想**不要了** → 说 **`撤销`**（扔掉本次未保存的改动）。

> 怎么判断你的意图：说「发版/提交/推送/上线」→ 提交；说「撤销/不要了/还原/改回去」→ 撤销。拿不准会直接问你。

---

### 4.1 发版（提交 + 推送） ｜ 关键字 `发版`

**做什么**：把当前改动写一句话说明、存档，再上传到团队仓库的 `main`（主干）。
**只做提交+推送**，不包含找文件夹/装依赖（那是「初始化」的事）。

步骤（助手自动完成）：
```bash
cd <项目根目录>
git status --short          # 1. 看一眼当前有哪些改动
git add -u                  # 2. 选中「已存在文件」的改动（不会带上新文件，避免误传 .env、日志等）
git commit -m "..."         # 3. 自动写一句中文说明
git push origin main        # 4. 上传到主干
git log -1 --stat           # 5. 给你看本次提交了哪些文件
```

**说明文字（commit message）规则**：自动归纳成一句中文。
- 纯改文字 → `文案：<概括>`
- 含代码 → `更新：<概括>`

**安全护栏**（即使在快捷模式下也遵守）：
- 只用 `git add -u`，**不用** `git add .`（避免把无关的新文件传上去）。
- 上传前确认当前在 `main` 分支，不在就停下来问你。
- 上传失败（比如有冲突）**不会硬推**，原样把报错给你，交开发处理。
- 如果改动**异常多**，或包含 `package.json`、`pnpm-lock.yaml`、`.env`、数据库脚本等敏感文件，会**先暂停告诉你**，不一键推上去。

**完成后你会看到**：提交了哪些文件、说明文字、上传结果。

> 「普通模式」（你没说发版，而是逐步让助手提交）：助手会**逐个文件**精确添加（`git add <具体文件>`），绝不会 `git add .` 一锅端；发现改动里夹着大量无关内容会停下来问你。说明文字建议用 `文案：` 开头。

---

### 4.2 撤销本次修改 ｜ 关键字 `撤销`

**做什么**：扔掉**本次还没提交**的改动，文件恢复成上次存档的样子。适合「改错了、不想要了、想重来」。

> ⚠️ **扔掉就找不回来了**，所以助手一定会先跟你确认。

步骤：
1. 先列出「将要被扔掉的改动」给你看：
   ```bash
   cd <项目根目录>
   git status --short          # 哪些文件改过/新增了
   git diff --stat             # 每个文件改了多少
   ```
2. 警告你：「以下改动将被**永久丢弃、无法找回**：<文件清单>。确认吗？」
3. 你明确说「确认/是的/撤销吧」之后才执行；你稍有犹豫就不动。
4. 执行：把改过的文件还原（`git restore <文件>`）；新增的文件**默认不删**，会单独问你要不要删。
5. 完成后告诉你「已撤销，已恢复原样」。

**重要边界**：
- 撤销**只对「还没提交」的改动**有效。
- 如果改动**已经发版（提交/上传）了**，助手**不会**去硬改远程仓库（那风险很高），而是告诉你「这版已经上传，没法简单撤销，请联系开发」。
- 不使用 `git reset --hard` 这种「连底裤都清掉」的危险操作。

---

## 安全底线（这些绝对不会做）

- ❌ 不强行推送（`push --force`）、不硬重置（`reset --hard`）、不删分支、不改历史。
- ❌ 不改依赖清单（`package.json`）、构建配置、数据库脚本、环境变量文件。
- ❌ 不把 `.env`、密钥、token 等敏感信息写进代码或上传。
- ❌ 不为了「修 bug」去动业务逻辑——本技能只管**起服务、改文字、提交/撤销**；遇到逻辑问题让你找开发。
- ✅ 任何拿不准的操作，**先问你再动手**。
